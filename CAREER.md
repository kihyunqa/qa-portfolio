## 두플 — QA 파트장

**2024.11 ~ 2025.02 · 3개월 · 반도체 공정 시뮬레이션 스타트업**

### 상황과 과제

입사 당시 QA 프로세스가 명문화되어 있지 않았습니다. 릴리즈 직전에 버그를 발견하거나, TC 리뷰 기준이 팀원마다 달라 품질이 일정하지 않은 상태였습니다. QA Part Leader로 합류하면서 가장 먼저 한 일은 "어떤 기준으로 릴리즈를 막을 것인가"를 팀 전체가 공유하는 것이었습니다.

### 내가 한 것

**릴리즈 게이트 기준 수립**  
P1·P2·P3 심각도 분류 기준을 정의하고 팀 내에 공유했습니다. "P1 버그가 열려 있으면 릴리즈를 막는다"는 규칙을 처음으로 명문화했고, 이 기준을 기반으로 릴리즈 전 TC 리뷰를 주도했습니다.

**TC 설계 프로세스 확립**  
릴리즈 사이클별로 테스트 범위를 정의하고, 우선순위 기반으로 TC를 설계하는 프로세스를 만들었습니다. 기존에는 개발 완료 후 즉시 테스트에 들어갔지만, 기능 명세 단계부터 QA가 개입하는 구조로 바꿨습니다.

### 결과

릴리즈 게이트 기준을 수립한 이후 P1 버그가 릴리즈에 포함된 케이스가 0건이었습니다. 스타트업 구조 변화로 3개월 후 팀이 해체됐지만, 이 경험에서 "QA는 테스트만 하는 역할이 아니라 릴리즈 기준을 세우는 역할"이라는 관점을 확립했습니다.

## IMS Mobility — QA 대리

**2022.03 ~ 2024.02 · 2년 · 모빌리티 플랫폼 (모바일앱/웹/백오피스)**

### 상황과 과제

모바일앱, 웹, 백오피스 세 영역을 동시에 QA해야 하는 환경이었습니다. 회귀 테스트가 매 릴리즈마다 반복됐고, 결제 도메인은 PG사 연동 시나리오가 복잡해 수동으로 전 케이스를 커버하기가 어려웠습니다. 특히 결제 금액 계산 로직은 소수점 처리 방식이 문서에 명시되지 않아 항상 리스크가 있었습니다.

### 내가 한 것

**Cypress E2E 자동화 도입**  
반복 회귀 테스트를 수동에서 자동화로 전환했습니다. 로그인·검색·결제·알림 핵심 플로우를 Cypress 스크립트로 작성하고 CI 파이프라인에 연동했습니다. 이 결정은 혼자 제안하고 설득한 것으로, 처음에는 "QA가 왜 코드를 짜느냐"는 반응이 있었지만 첫 릴리즈에서 회귀 테스트 시간이 눈에 띄게 줄면서 이후에는 자연스럽게 정착됐습니다.

**결제 QA 전담 — P1 에스컬레이션 주도**  
PG사 연동 결제 시나리오를 전 케이스 커버했습니다(정상·실패·취소·환불·부분결제). 테스트 중 다중 상품에 할인 쿠폰을 조합했을 때 소수점 오차가 발생하는 케이스를 발견했습니다. 단순 표시 오류가 아니라 실제 결제 금액 불일치로 이어질 수 있는 문제였기 때문에 P1으로 판단하고 에스컬레이션을 주도했습니다. 릴리즈 일정이 이틀 남은 상황이었지만 수정을 설득해 출시 전에 완료됐습니다.

**Swagger 기반 API 검증**  
REST API 30+ 엔드포인트를 GET/POST/PUT/PATCH/DELETE 전 메서드, 인증 토큰 포함해 검증했습니다. API 문서와 실제 동작의 불일치를 발견하는 것이 목적이었고, 이 과정에서 응답 스펙이 문서화되지 않은 케이스를 수차례 개발팀에 피드백했습니다.

### 결과

결제 오차 P1 에스컬레이션은 출시 후 고객 민원으로 이어질 수 있었던 케이스를 사전에 막은 경험입니다. Cypress 도입으로 회귀 테스트 반복 공수가 줄었고, API 검증 커버리지를 30+ 엔드포인트까지 확장했습니다.

## 모비프렌 (삼성전자 파트너) — QA 주임

**2017.09 ~ 2022.01 · 4년 4개월 · 삼성전자 파트너사**

### 상황과 과제

삼성전자 파트너사로서 SmartThings IoT 플랫폼, AI Bixby, Galaxy 모바일앱을 검증하는 역할이었습니다. 디바이스 종류가 50개 이상이고 프로토콜이 Wi-Fi·BLE·Z-Wave로 다양했기 때문에, "이 디바이스가 새 펌웨어에서 여전히 잘 작동하는가"를 체계적으로 검증하는 프로세스가 필요했습니다. Bixby 음성 명령의 경우 엣지케이스 설계가 특히 까다로웠는데, 소음 환경이나 억양 변형 같은 실사용 조건을 테스트 시나리오로 구조화하는 작업이 주된 과제였습니다.

### 내가 한 것

**IoT SmartThings 검증 — 50+ 디바이스**  
Wi-Fi·BLE·Z-Wave 프로토콜별 연동 시나리오를 설계하고 신규 디바이스 온보딩 QA 프로세스를 표준화했습니다. 단순히 "연결이 되는가"가 아니라, 앱에서 제어했을 때 실제 디바이스가 의도한 대로 동작하는지, 연결 해제 후 재연결 시 상태가 복원되는지까지 확인했습니다.

**AI Bixby 음성 명령 검증**  
음성 인식률, 명령 시나리오 흐름, 엣지케이스 대규모 검증을 담당했습니다. 한국어/영어 인식 정확도 비교 테스트와 함께, 소음 환경·일정 거리 이상·억양 변형 등 실사용 조건을 반영한 시나리오를 직접 설계했습니다. 인식 오류 유형을 분류하는 기준을 만들어 개발팀 피드백의 명확성을 높였습니다.

**삼성 모바일앱 QA**  
Galaxy 플래그십 전 모델 회귀 테스트 및 OS 업데이트 기준 기능 검증을 수행했습니다.

### 결과

4년 4개월 동안 SmartThings·Bixby·Galaxy 전 영역에 걸쳐 QA를 담당했습니다. 이 기간 동안 IoT 특유의 하드웨어-소프트웨어 경계 버그, AI 음성 명령의 엣지케이스 패턴, 플래그십 모델 회귀 테스트 설계 등 다양한 QA 도메인을 경험했습니다.

## Claude MCP 자동화 포트폴리오 (현재 구축 중)

**2025 ~ 현재 · 개인 프로젝트**

### 왜 시작했는가

QA 현장에서 반복적으로 느낀 문제가 있었습니다. TC를 작성하고, 결과를 문서화하고, 팀에 알리고, 이슈 트래커에 연결하는 작업이 모두 수동으로 이루어지고 있었습니다. "이 반복 작업을 자동화할 수 있다면 QA가 정말 집중해야 할 것에 더 많은 시간을 쓸 수 있겠다"는 생각에서 Claude MCP 연동을 시작했습니다.

### 내가 한 것

**MCP 5개 실제 연동 — 코드 0줄**  
- filesystem MCP: TC 파일 자동 생성·저장·읽기  
- playwright MCP: E2E 브라우저 테스트 자동 실행, 스크린샷 캡처  
- github MCP: 레포 커밋·PR·파일 업로드 대화로 처리  
- notion MCP: TC 결과 Notion 페이지 자동 문서화 (실제 연동 확인)  
- slack MCP: QA 완료 알림 Slack 채널 자동 발송 (실제 연동 확인)  

**TC 자동생성 파이프라인**  
기능 명세를 입력하면 Claude가 해피패스·네거티브·엣지케이스를 분류해 TC를 자동 생성합니다. 현재 누적 145건+이며, 로그인·결제·보안·검색·API·접근성·성능 전 카테고리를 커버합니다. 이 과정에서 코드를 한 줄도 작성하지 않았습니다.

**Playwright E2E 12개 spec**  
login·search·cart·api·payment·security·signup·notification·performance·accessibility 등 12개 spec. GitHub Actions 2개 운영(TC push → Slack 자동 알림).

**Jira 연동**  
GitHub for Atlassian 연동으로 커밋 메시지 기반 Jira 이슈가 자동 트래킹됩니다.

### 결과

TC 생성부터 Slack 알림까지 7단계 파이프라인을 대화 한 번으로 실행할 수 있게 됐습니다. "코드를 짜지 않아도 QA 자동화를 구현할 수 있다"는 것을 직접 증명한 프로젝트이며, 동시에 6.9년 QA 현장 경험이 있어야 "무엇을 자동화할지" 판단할 수 있다는 것도 확인한 경험입니다.
