> 6년 9개월의 QA 경력과 Claude MCP 실제 도입 경험을 바탕으로 작성한 비전 문서

---

## 왜 AI QA인가

QA 엔지니어가 하루에 소비하는 시간의 상당 부분은 **반복 작업**이다.

- TC 문서 작성 및 정리
- 테스트 결과 Notion/스프레드시트에 업로드
- 슬랙/이메일로 팀에 완료 공유
- GitHub에 파일 커밋 및 PR 생성

이 작업들은 중요하지만, QA 엔지니어의 **핵심 가치인 "무엇을 테스트할 것인가"** 에 집중할 시간을 빼앗는다.

Claude MCP는 이 반복 구간을 자동화하여 QA가 **판단과 설계에 집중**할 수 있게 만든다.

---

## 전통적 QA vs AI 기반 QA (실제 비교)

| 항목 | 전통적 QA | AI 기반 QA (MCP) |
|------|-----------|------------------|
| TC 작성 | 수동 2시간+ | Claude 대화로 5분 |
| 문서화 | 수동 업로드 | Notion MCP 자동 동기화 |
| 코드 커밋 | 수동 push | GitHub MCP 자동 커밋 |
| 실패 알림 | 수동 커뮤니케이션 | Slack MCP 실시간 발송 |
| E2E 실행 | 스케줄링 필요 | CI/CD + Playwright 자동 트리거 |
| 보고서 작성 | 템플릿 수동 채우기 | 결과 자동 집계·정리 |
| 회귀 테스트 | 릴리스마다 수동 실행 | PR 생성 시 자동 트리거 |
| 버그 등록 (예정) | 수동 Jira 입력 | MCP → 자동 이슈 생성 |

## 이 포트폴리오에서 실제로 검증한 것

### Phase 1 — 완료 ✅

```
사용자(QA): "로그인 TC 만들어줘, GitHub에 올리고, Notion에 정리하고, Slack으로 알려줘"
Claude MCP:
  1. filesystem → TC 파일 로컬 생성
  2. github    → 자동 커밋 (testcase_login.md)
  3. notion    → TC 허브 페이지 자동 업데이트
  4. slack     → 새-채널 완료 알림 발송
  5. playwright→ E2E 브라우저 자동화 spec 실행
```

**실제 달성 수치:**

| 항목 | 결과 |
|------|------|
| TC 생성 | 145건+ (16개 파일) |
| Playwright spec | 12개 (9개 영역 커버) |
| GitHub 커밋 | 80개+ 파일 자동 관리 |
| Notion 업로드 단축 | 30분 → 30초 (98% 절감) |
| GitHub Actions 파이프라인 | 2개 (qa-notify + playwright) |
| 코드 직접 작성 | 0줄 — 대화만으로 구현 |

### Phase 2 — 예정 🔜

```
Jira MCP 연동:
  버그 발견 → Claude가 Jira 이슈 자동 등록 → 담당자 자동 배정 → Slack 알림
```

## 2026년 AI QA 트렌드와 나의 포지션

> interview-qa-advanced.md Q11 연계: "AI 시대에 QA 엔지니어의 역할은 어떻게 바뀌는가"

### 업계 변화 흐름

| 트렌드 | 내용 | 이 포트폴리오의 대응 |
|--------|------|---------------------|
| LLM 기반 테스트 생성 | AI가 TC 초안 작성 | Claude MCP로 실제 구현 ✅ |
| Agent 기반 자동화 | AI가 멀티스텝 작업 수행 | MCP 5개 체이닝으로 검증 ✅ |
| 자연어 → 코드 | 비개발자도 자동화 가능 | 코드 0줄 대화만으로 구현 ✅ |
| CI/CD 통합 심화 | PR → 자동 테스트 트리거 | GitHub Actions 2개 구축 ✅ |
| 실시간 품질 모니터링 | 실패 즉시 알림 | Slack MCP 연동 ✅ |

### AI 시대 QA 엔지니어의 핵심 역량 (변화하는 것과 변하지 않는 것)

**AI가 대체하는 것:**
- 단순 반복 TC 문서화
- 정형화된 회귀 테스트 실행
- 결과 취합 및 보고서 작성

**AI가 대체할 수 없는 것:**
- 제품에 대한 깊은 이해와 리스크 판단
- 사용자 관점에서의 탐색적 테스트
- AI 출력물의 품질 검수 (QA of AI)
- 비즈니스 임팩트 기반 우선순위 결정

## QA 엔지니어에게 AI가 바꾸는 것

### 바뀌는 것
- 반복 문서화 작업 → 자동화
- 팀 공유 타이밍 → 실시간 자동 알림
- TC 작성 속도 → 대화 기반으로 대폭 단축
- 커밋·PR → MCP가 처리

### 바뀌지 않는 것
- **무엇을 테스트할지 판단하는 능력**
- **엣지케이스와 리스크를 감지하는 QA 감각**
- **팀과 제품을 이해하는 도메인 지식**
- **우선순위 판단 — P0/P1 트리아지 능력**

AI는 QA를 대체하지 않는다.  
**반복 구간을 AI에게 맡기고, QA는 본질에 집중한다.**

## 이 경험이 실무에 주는 의미

이 포트폴리오는 단순한 예제가 아니라 **실제 작동하는 QA 자동화 파이프라인**이다.

- QA 도구를 쓰는 것과 **QA 파이프라인을 설계·구축**하는 것은 다르다
- MCP를 통해 "대화 → 자동화"의 전체 흐름을 직접 검증했다
- 실무 도입 시 팀 규모·프로젝트 특성에 맞춰 MCP 구성 커스터마이징 가능
