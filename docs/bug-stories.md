## STORY-001 · 로그인 세션 토큰이 로그아웃 후에도 유효한 문제

### 상황
회귀 테스트 중 로그아웃 플로우를 검증하던 중 발견.  
로그아웃 버튼 클릭 → 메인 화면 리다이렉트가 정상으로 보였으나,  
**로그아웃 직전 복사해둔 Authorization 헤더 토큰으로 API 재호출 시 200 OK 반환됨.**

### 발견 과정
```
1. Postman으로 로그인 → Bearer 토큰 복사
2. UI에서 로그아웃 실행
3. 복사한 토큰으로 /api/user/profile 직접 호출
4. → 200 OK + 사용자 정보 정상 반환 ← 이상
```

### QA 판단 포인트
- UI 로그아웃은 단순히 localStorage 토큰 삭제 + 리다이렉트였음
- 서버 측 토큰 무효화(블랙리스트 처리 또는 만료 단축)가 구현 안 된 상태
- 이 상태로 출시하면 **공용 PC 환경, 탈취된 토큰으로 계정 접근 가능**

### 영향도 분석
| 구분 | 내용 |
|------|------|
| 심각도 | P1 (보안) |
| 영향 범위 | 전체 로그인 사용자 |
| 재현율 | 100% |
| 출시 차단 여부 | **차단 권고** |

### 처리 결과
- 개발팀에 서버 사이드 토큰 무효화 구현 요청
- 임시 조치로 토큰 만료 시간 30분 → 5분으로 단축
- 이후 Redis 블랙리스트 방식으로 최종 해결

### QA 인사이트
> 로그아웃은 UI 동작만 확인하면 통과처럼 보이지만,  
> **실제 보안은 서버 측에서 토큰을 죽이는지**가 핵심입니다.  
> 체크리스트에 "로그아웃 후 이전 토큰으로 API 호출 차단 확인" 항목을 추가했습니다.

## STORY-002 · 결제 금액 계산 시 소수점 버림으로 인한 1원 오차 누적

### 상황
결제 모듈 신규 개발 후 QA 진행 중 발견.  
단일 상품 결제는 정상이었으나, **3개 이상 상품 혼합 구매 + 할인율 적용 시**  
최종 결제 금액이 장바구니 합산 금액과 1원 차이 발생.

### 발견 과정
```
케이스: 상품 A(13,333원) × 2 + 상품 B(9,999원) × 1, 10% 할인 적용

기대값: (13,333 × 2 + 9,999) × 0.9 = 32,999원
실제값: 33,000원 (1원 초과 청구)

원인: 각 상품별 할인금액을 먼저 계산 → 소수점 truncate → 합산
     → 상품별로 버려진 소수점이 누적
```

### QA 판단 포인트
- 1원 차이는 작아 보이지만, **대량 거래 시 수천 건 × 1원 = 수천 원 오과금**
- 사용자 민원 + 전자금융거래법상 오과금 이슈 가능
- 개발팀이 단순 반올림 이슈로 취급하려 했으나, **계산 순서 자체의 설계 문제임을 설명**하고 P1으로 에스컬레이션

### 영향도 분석
| 구분 | 내용 |
|------|------|
| 심각도 | P1 (금전 오류) |
| 영향 범위 | 다중 상품 + 할인 동시 적용 케이스 |
| 재현율 | 특정 금액 조합에서 70%+ |
| 출시 차단 여부 | **차단 권고** |

### 처리 결과
- 계산 방식을 "상품별 계산 후 합산"에서 "전체 합산 후 할인 1회 적용"으로 변경
- 소수점 처리 정책(round vs floor vs ceil) 개발/기획/QA 3자 합의 문서화
- 경계값 TC 12건 추가 작성 (소수점이 발생하는 금액 조합)

### QA 인사이트
> 결제 TC는 **경계값과 조합**이 핵심입니다.  
> 단일 케이스 통과 = 정상이라는 함정에 빠지기 쉽고,  
> "의심스러운 케이스를 끝까지 파고드는 습관"이 이 버그를 잡았습니다.

## STORY-003 · 알림 중복 발송 — 특정 네트워크 환경에서만 재현

### 상황
QA 사이클 종료 후 스테이징 환경에서 알림 기능 검증 중 발견.  
정상 환경에서는 1건 발송이었으나,  
**네트워크 지연 시뮬레이션(throttling) 환경에서 동일 이벤트에 알림 2~3건 중복 발송.**

### 발견 과정
```
1. Chrome DevTools → Network → Slow 3G 설정
2. 특정 액션(주문 완료) 실행
3. 알림 수신 확인 → 동일 내용 3건 수신
4. 일반 네트워크 → 재현 안 됨
5. → 클라이언트 retry 로직 + 서버 멱등성 미처리 조합으로 발생
```

### QA 판단 포인트
- 일반 테스트에서는 절대 발견 안 됨 → **조건을 의도적으로 나쁘게 만드는 네거티브 테스트**가 핵심
- 재현 조건이 까다로워 "간헐적 이슈"로 묻힐 뻔했으나, 재현 환경(Slow 3G)을 문서화해 개발팀 전달
- **실 사용자의 지하철, 엘리베이터 환경과 동일한 조건**임을 근거로 P2 등록

### 영향도 분석
| 구분 | 내용 |
|------|------|
| 심각도 | P2 (UX 품질) |
| 영향 범위 | 모바일 환경 저속 네트워크 사용자 |
| 재현율 | Slow 3G 설정 시 80%+ |
| 출시 차단 여부 | 조건부 허용 (핫픽스 플랜 병행) |

### 처리 결과
- 서버 측 알림 이벤트 중복 처리 방지 (idempotency key 적용)
- 클라이언트 retry 로직에 지수 백오프(exponential backoff) 추가
- 알림 발송 로그 모니터링 대시보드 추가 요청

### QA 인사이트
> "재현이 어렵다"는 건 "테스트 조건이 좁다"는 뜻입니다.  
> 실 사용자 환경을 시뮬레이션하는 것이 QA의 역할이고,  
> 재현 조건을 문서화하는 것이 개발팀과의 신뢰를 만드는 방법입니다.

## 🧠 공통 QA 원칙 (이 케이스들에서 도출)

1. **UI 통과 ≠ 서버 통과** — 항상 API 레벨까지 확인
2. **경계값과 조합** — 단일 케이스만 보면 절반도 못 잡음
3. **나쁜 조건을 의도적으로 만들어라** — 좋은 환경에서만 테스트하면 현실 버그를 못 잡음
4. **심각도 판단은 데이터로** — "1원이잖아요"에 흔들리지 않으려면 영향 건수 × 금액으로 설명
