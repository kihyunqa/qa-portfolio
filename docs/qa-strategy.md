# 🎯 QA Strategy Document

> MCP 기반 자동화 포트폴리오 — QA 전략 및 방법론

---

## 1. QA 전략 개요

이 포트폴리오는 **MCP(Model Context Protocol) 5개 도구**를 활용한 AI 기반 QA 자동화를 핵심으로 합니다.

```
[QA 전략 피라미드]

        /\
       /  \         E2E Tests (10%)
      /----\
     /      \       Integration Tests (20%)
    /--------\
   /          \     Unit + API Tests (70%)
  /____________\
```

---

## 2. 테스트 범위 (Scope)

### In Scope
- 인증/인가 기능 (로그인, 세션, 권한)
- 핵심 비즈니스 로직 (검색, 장바구니, 결제)
- API 엔드포인트 (REST API 전체)
- 크로스 브라우저 호환성
- 접근성 (WCAG 2.1 AA)
- 성능 (LCP < 2.5s, FID < 100ms)

### Out of Scope
- 서드파티 결제 게이트웨이 내부 로직
- 하드웨어 의존 기능
- 레거시 IE 브라우저

---

## 3. MCP 도구별 역할 분담

| MCP 도구 | 역할 | 트리거 |
|---------|------|--------|
| **filesystem** | TC 파일 생성/관리, 로컬 리포트 저장 | 수동/자동 |
| **github** | 코드 리뷰, PR 자동 체크, Actions 실행 | Push/PR 이벤트 |
| **playwright** | E2E 브라우저 자동화 테스트 | 일정/PR |
| **notion** | TC 결과 문서화, 대시보드 업데이트 | 테스트 완료 후 |
| **slack** | QA 결과 알림, 버그 발견 즉시 통보 | 이벤트 기반 |

---

## 4. 테스트 사이클

```
[개발 사이클과 QA 통합]

Dev 코드 작성
     ↓
PR 생성 → GitHub Actions 트리거
     ↓
Playwright E2E 자동 실행
     ↓
결과 → Slack 알림 발송
     ↓
통과 → Notion 문서화
     ↓
머지 → 릴리스 QA 완료
```

---

## 5. 버그 관리 프로세스

1. **발견** → 버그 리포트 작성 (bug-report-template.md 양식)
2. **심각도 분류** → P0~P3 기준 적용
3. **Slack 즉시 알림** → Critical은 @channel 태그
4. **GitHub Issue 생성** → 담당자 지정
5. **수정 후 재테스트** → 회귀 테스트 포함
6. **Notion 이력 기록** → 버그 히스토리 누적

## 6. 릴리스 QA 게이팅 기준

| 조건 | 기준 | 미충족 시 |
|------|------|----------|
| P0 버그 | 0건 | 릴리스 블로킹 |
| P1 버그 | 0건 | 릴리스 블로킹 |
| 자동화 테스트 통과율 | 98% 이상 | 릴리스 블로킹 |
| 접근성 점수 | AA 등급 이상 | 권고 (블로킹 아님) |
| 성능 점수 | Lighthouse 80+ | 권고 |

---

## 7. 지속적 개선 (Continuous Improvement)

- **매주 금요일**: QA 메트릭 리뷰 (통과율, 커버리지, 버그 추이)
- **매월**: 자동화 스크립트 리팩토링
- **분기별**: TC 전체 검토 및 불필요 TC 정리
- **신기능 출시 전**: 리스크 기반 테스트 계획 수립
