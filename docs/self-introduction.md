## 2. 직무 관련 경험 및 역량

```
6년 9개월간 세 회사에서 QA 엔지니어로 근무하며 다음 역량을 쌓았습니다.

[ TC 설계 ]
해피패스·네거티브·엣지케이스를 포함한 체계적인 테스트케이스 설계 경험.
현재까지 총 145건+ TC를 직접 설계하고 관리했습니다.

[ E2E 자동화 ]
IMS Mobility에서 Cypress를 활용한 E2E 자동화를 구축했으며,
최근에는 Playwright MCP를 통해 12개 spec 파일(로그인·결제·보안·
접근성·성능·알림 등 전 영역)을 작성하고 실행까지 완료했습니다.

[ API 테스트 ]
Swagger 기반 RESTful API 검증 경험. 요청/응답 구조, 엣지케이스,
에러 핸들링까지 체계적으로 검증하는 TC를 설계했습니다.

[ CI/CD 연동 ]
GitHub Actions 2개 워크플로우를 직접 설계. TC 파일 커밋 감지 →
Slack 자동 알림까지 QA-CI 연동 파이프라인을 구축했습니다.

[ AI 도구 활용 ]
Claude MCP 5개(filesystem, github, playwright, notion, slack)를
직접 연동해 QA 자동화 파이프라인을 구축했습니다. 코드 한 줄 없이
자연어 지시만으로 전 과정을 자동화했습니다.
```

## 3. 성격의 장단점

```
[ 장점: 꼼꼼함과 체계적 문서화 ]

저의 가장 큰 강점은 "기록하는 습관"입니다.
테스트 결과, 버그 리포트, QA 전략 문서까지 — 제가 진행한 모든 작업은
다음 사람이 이어받을 수 있도록 기록으로 남깁니다.

실제로 이번 포트폴리오 프로젝트에서 수십 번의 세션을 이어가면서
NEXT_STEPS.md와 CHANGELOG.md를 인수인계 문서로 활용해,
작업의 연속성을 유지했습니다. 이 습관은 팀 환경에서 QA 부채를 줄이고
온보딩 비용을 낮추는 데 직접적으로 기여합니다.

[ 단점: 완성도에 대한 집착 ]

때로는 "충분히 좋은" 수준에서 멈추는 것이 어렵습니다.
TC 하나를 작성할 때도 빠진 케이스가 없는지 반복해서 검토하는 편입니다.
이를 보완하기 위해 타임박스(시간 제한)를 정해두고 작업하는 방식을
의식적으로 훈련하고 있습니다.
```
